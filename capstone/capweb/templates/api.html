{# ==============> OVERVIEW <============== #}

The Caselaw Access Project API, also known as CAPAPI, serves all official, published US court cases, and their related metadata, from 1658 to 2018. We digitized every case in this database during the first phase of the Caselaw Access Project, which we embarked upon in 2013. Before that, nearly every case was sitting in its original binding on Harvard Law Library shelves. Click here to learn more about the project.

CAPAPI is RESTful, and serves most of its data in JSON, but case text is also available in plain text, HTML, and XML. Though APIs are primarily useful to developers, anybody with a web browser can interact with it, though the interface is a bit clunky. (Perhaps, one day, some of you ambitious software developers in the audience will make a lovely, open-source case browsing application using this API. *nudge* *nudge*) Click here for some usage examples.

Our goal in undertaking this project was to make the data as open as possible. While there are restrictions that affect some users trying to access more than 500 cases per day, metadata for every case in every jurisdiction, and all cases from Illinois and Arkansas are available without restriction; you don't even need to be logged in or use an API key. Click here for more information on access restrictions, what type of users aren't subject to them, and how you can advocate to eliminate them in your legal jurisdiction.

{# ==============> USING CAPAPI <============== #}

Basic CAPAPI Usage

Click here to visit the API in your web browser.

CAPAPI is a RESTful API based on Django Rest Framework.. This means it can be accessed with standard web tools, like curl, http request libraries in your favorite programming language, and even web browsers. If you're already feeling lost, you might want to pick up some of the background information we've put into the beginner's introduction.

The return format for most of the data is JSON, but cases are available in HTML, and their original XML. In addition, the cases are avaible with

Authenticating to retrieve non-whitelisted cases is simple. If you're using a web browser, just log in to the site. Programmatically, or from the command line, just pass the "Authorization: Token your-token-here" header in your request. For example, if your authentication token is abcd12345, a curl command would look like this:

curl -H "Authorization: abcd12345" "{{ request.build_absolute_uri }}api/v1/cases/?limit=1&full_case=true"

The API defaults to returning one hundred items per response. If you want more than one hundred items, you can page through subsequent items using the offset parameter. For example, if you want the second page of one hundred items, set the offset value to 100. For convenience, the API response includes a next key containing the URL to the next page of results.

{# ==============> BEGINNERS <============== #}

Beginner's Introduction

Are you a little lost in all the technical jargon? Here's a good place to start. This is by no means a complete introduction to using APIs, but it might be just enough to help situate a technically inclined person who's a bit outside of their comfort zone.

Fundamentally, an API is no different from a regular website: A program on your computer sends a bit of data to a server, the server processes that data, and then sends a response. If you know how to read a URL, you can interact with web-based services in ways that aren't limited to clicking on the links and buttons on the screen.

Consider the following URL, which will perform a google search for the word "CAP." 

https://www.google.com/search?q=CAP

Let's break it down into its individual parts: 

https://

This first part tells your web browser which protocol to use: this isn't very important for our purposes, so we'll ignore it. 

www.google.com

The next part is a list of words, separated by periods, between the initial double-slash, and before the subsequent single slash. Many people refer to this as the domain, which is only partly true; the important takeaway here is that it points to a specific server, which is just another computer on the internet. 

/search

The next section, which is comprised of everything between the slash after the server name and the question mark, is called the path. It's called a path because, in the earlier days of the web, it was a path to a specific file on the web server. These days, it's more likely that the path will point to a specific endpoint.

You can think of an endpoint as a distinct part of a program, which could require specific inputs, and/or provide different results. For example, the "login" endpoint on a website might accept a valid username and a password for input, and return a message that you've successfully logged in. A "register" endpoint might accept various bits of identifying information, and return a screen that says your account was successfully registered.

Though there is only one part of this particular path, search, developers usually organize paths into hierarchical lists separated by slashes. Hypothetically, if the developers at Google decided that one generalized search endpoint wasn't sufficiently serving people who wanted to search for books or locations, they could implement more specific endpoints such as /search/books and /search/locations. 

?q=CAP

The final section of the URL is where you'll find the parameters, and is comprised of everything after the question mark. Parameters are a way of passing individual pieces of data, or values, to the endpoint to help it perform its job. In this case, the parameter tells the /search endpoint what to search for. Without this parameter, the response wouldn't be particularly useful.

A URL can contain many parameters, separated by ampersands, but in this instance, there is only one parameter: the rather cryptically named "q," which is short for "query," that has a value of "CAP." Parameter names are arbitrary— Google's developers could just as easily have set the parameter name to ?query=CAP, but decided that "q" would sufficiently communicate its purpose.

The Google developers designed their web search endpoint to accept other parameters, too. For example, there is an even more cryptically named parameter, 'tbs' which will limit the age of the documents returned in the search results. The parameters ?q=CAP&tbs=qdr:y will perform a web search for "CAP" and limit the results to documents less than a year old.

So when you're working with CAPAPI, the same principles apply. Rather than http://www.google.com, you'll be using {{ request.build_absolute_uri }}. Rather than using the /search?q= endpoint and parameter, you'll be using one of our endpoints and the parameters we've defined. One important difference is the purpose of the structured data we're returning, vs. the visual, browser-oriented data that google is returning with their search engine.

When you perform a query in a web browser using our API, there are some links and buttons, but the data itself is in a text-based format with lots of brackets and commas. This format is called JSON, or JavaScript Object Notation. We use this format because software developers can easily utilize data in that format in their own programs. We do intend to have a more user-friendly case browser at some point soon, but we're not quite there yet.

OK! That about does it for our beginner's introduction to web-based APIs. Please check out our usage examples section to see some of the ways you can put these principles to work in CAPAPI. If you have any suggestions for making this documentation better, we'd appreciate your taking the time to let us know in an issue report in our code repository on github.com.

Thanks, and good luck!

{# ==============> EXAMPLES <============== #}

Usage Examples

This is a non-exhaustive set of examples intended to orient new users. The endpoints section contains more comprehensive documentation about the URLs and their parameters.

{# ====> single case <==== #}
Retrieve a single case, by its ID
{{ request.build_absolute_uri }}api/v1/cases/{{ case_id }}/

This example uses the single case endpoint, and will retrieve the metadata for a single case.

Modification with Parameters:
  • With HTML-formatted Case Body and Metadata
    • {{ request.build_absolute_uri }}api/v1/cases/{{ case_id }}/
    • To ingest a JSON object with both metadata and a field containing an easily stylable HTML case body, set the body_format parameter to html. Set it to xml to get the case body with its original markup.
  • Raw original XML document, with METS data
    • {{ request.build_absolute_uri }}api/v1/cases/{{ case_id }}/?full_case=true&format=xml
    • To get the document by itself, without the JSON enclosure and metadata, use the format parameter. Set it to HTML and get a display-ready, standalone HTML document.
{# ====> filter cases <==== #}
Retrieve a list of cases by citation
{{ request.build_absolute_uri }}api/v1/cases/?cite={{ case_metadata.citations.0.cite }}

This example uses the cases endpoint, and will retrieve every case with the citation {{ case_metadata.citations.0.cite }}.

There are many parameters with which you can filter the cases result. Check the cases endpoint documentation for a complete list of the parameters, and what values they accept.

Modification with Parameters:
  • Add a date range filter
    • {{ request.build_absolute_uri }}api/v1/cases/?cite={{ case_metadata.citations.0.cite }}&decision_date_min=1900-12-30&decision_date_max=2000-06-15
    • You can combine any of these parameters to refine your search. Here, we have the same search as in the first example, but will only recieve results from within the specified dates.
  • Raw original XML document, with METS data
    • {{ request.build_absolute_uri }}api/v1/cases/{{ case_id }}/?full_case=true&format=xml
    • To get the document by itself, without the JSON enclosure and metadata, use the format parameter. Set it to HTML and get a display-ready, standalone HTML document.
{# ====> Paginate <==== #}
Traversing Pages of Results
{{ request.build_absolute_uri }}api/v1/cases/?offset=100

All queries in CAPAPI default to, and are limited to 100 results. This query will still return 100 results, but it will skip 100, so you'll start at 101 rather than 1, and get the subsequent 100 results.

Modification with Parameters:
  • Get smaller list of results
    • {{ request.build_absolute_uri }}api/v1/cases/?offset=20&limit=10
    • Combining the offset and limit parameters, you can get a customize your pagination to fit your needs. This URL will recieve 10 results starting at the 20th result.
{# ====> get reporters for a jurisdiction <==== #}
Get all reporters in a jurisdiction
{{ request.build_absolute_uri }}/api/v1/reporters/?jurisdictions=ark

This example uses the reporter endpoint, and will retrieve all reporters in Arkansas.

Case Browse/Search Endpoint

This is the primary endpoint; you use it to browse, search for, and retrieve cases. If you know the numeric ID of your case in our system, you can append it to the path to retrieve a single case.

Endpoint Parameters:
  • name_abbreviation
    • An arbitrary string
    • e.g. People v. Smith
  • decision_date_min
    • YYYY-MM-DD
  • decision_date_max
    • YYYY-MM-DD
  • docket_number
    • An arbitrary string
  • citation
    • e.g. 1 Ill. 21
  • reporter
    • integer
    • a reporter id
  • court
    • slug
    • a court slug
  • jurisdiction
    • slug
    • a jurisdiction slug
  • search
    • An arbitrary string
    • A full-text search query
  • offset
    • integer
    • The result offset. By default, CAPAPI will return 100 results for your query. If your query yielded 500 results and the offset parameter was not set, your code would return result 1-101. If you set offset=100 then you would get the second set of 100.
  • limit
    • integer
    • The result offset. By default, CAPAPI will return 100 results for your query. If set lower than 100, the limit parameter will limit the number of results to your specification.

{# ==============> RESTRICTIONS <============== #}

Restrictions

The agreement with our project partner, Ravel, stipulates that only bonafide academic researchers can access more than 500 cases per day, so all registered users are subject to this restriction. However, this stipulation does not apply to jurisdictions with digital-first publishing, that is, they make their cases freely available online in a citable, machine-readable format. We call these whitelisted jurisdictions. Currently, Illinois and Arkansas are the only whitelisted jurisdictions.

We would love to whitelist more jurisdictions! If you are involved in US case publishing, we'd love to talk to you about going digital-first. Please send us an email at lil@law.harvard.edu and introduce yourself!

If you think you qualify as a bonafide academic researcher and would like unrestricted access to non-whitelisted cases, email us at lil@law.harvard.edu.

If you are interested in licensing this data for commercial use, Click here to contact Ravel for more information.

Glossary

This is a list of technical or project-specific terms we use in this docuemntation. These are stub definitions to help you get unstuck, but they should not be conisdered authoratative or complete. A quick Google search should provide more context for any of these terms.

curl
curl is a simple command line tool for retrieving data over the internet. It's similar to a web browser in that it will retrieve the contents of a url, but it will dump the text contents to a terminal, rather than show a rendered version in a graphical browser window.
Command Line
This is the textual interface for interacting with a computer. Rather than interacting with the system through windows and mouse clicks, commands are typed and output is rendered in its textual form. On mac, the default Command Line program is Terminal. On Windows, the program is cmd.exe.
URL
A URL, or Uniform Resource Locator, is an internet address that generally contains a communication protocol, a server name, a path to a file or endpoint, and possibly parameters to pass to the endpoint.
API
API is an acronym for Application Programming Interface. Broadly, it is a way for one computer program to transfer data to another computer program. CAPAPI is a RESTful API designed with the intent of distributing court case data.
RESTful
A RESTful API is based on HTTP, and makes use of its built-in verbs(commands), such as GET and POST.
Server
A server is just a computer on the internet that was configured to respond to requests from other computers. A web server will respond to requests from a web browser. An email server will respond to requests from email programs, and or other email servers which are sending it messages.
Endpoint
You can think of an endpoint as a distinct part of a program, which could require specific inputs, and/or provide different results. For example, the "login" endpoint on a website might accept a valid username and a password for input, and return a message that you've successfully logged in. A "register" endpoint might accept various bits of identifying information, and return a screen that says your account was successfully registered.
URL Parameter
For our purposes, a parameter is just a piece of data with a label that can be passed to an endpoint in a web request.

A string, as a type of data, just means an arbitrary (or string) of characters. A word is a string. This whole sentence is a string. "h3ll0.!" is a string.
Character
A letter, number, space, or piece of punctuation. Multiple characters together make up a string.
Special Characters
Characters that have significance to a program. For example, URLs can't contain certain pieces of punctuation. Whether any given character is deemed 'special' depends on the context.
Slug
A string with special characters removed for ease of inclusion in a URL.
